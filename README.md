# Incident Copilot

> A RAG (Retrieval-Augmented Generation) system for SRE and Platform Engineering that reduces MTTR by providing instant, context-aware answers about past incidents, runbooks, and alerts.

![CI](https://github.com/kgbunshin/incident-copilot/actions/workflows/ci.yml/badge.svg)
![Deploy](https://github.com/kgbunshin/incident-copilot/actions/workflows/deploy.yml/badge.svg)

---

## What it does

Incident Copilot ingests your operational knowledge — runbooks, post-mortems, logs, and Alertmanager alerts — into a local vector database. When an incident happens, you ask a question in plain English and get a relevant, source-backed answer generated by a local LLM.

**No data leaves your infrastructure.** Everything runs on your own servers.

```
You: "payment-service is OOMKilled, what should I do?"

Copilot: "Based on a similar incident on 2024-01-14, the root cause was a
         memory leak introduced in v2.3.1. Recommended: check recent deploys,
         consider rollback, and increase memory limit to 512Mi."

Sources: postmortem-2024-01-14.md, runbook-oom-k8s.md  |  Confidence: 87%
```

## Architecture

```
VPS 1 (k3s + Alertmanager)        VPS 2 (Incident Copilot)
──────────────────────────         ──────────────────────────────────
Alertmanager ──webhook──►  FastAPI :8080
                                    │
Grafana                             ├── POST /ingest/file   (manual upload)
                                    ├── POST /ingest/alert  (Alertmanager webhook)
                                    ├── POST /query         (RAG query)
                                    └── GET  /health

                                    ┌──────────┴───────────┐
                                 ChromaDB :8000         Ollama :11434
                                 (vector store)         (LLM + embeddings)
```

**Ingestion pipeline:** file/alert → parser → chunker → embedder (nomic-embed-text) → ChromaDB

**Query pipeline:** question → embed → ChromaDB top-k search → Mistral 7B → answer + sources

## Tech Stack

| Layer | Technology | Reason |
|-------|-----------|--------|
| LLM | Ollama + Mistral 7B Q4_K_M | Best quality/cost ratio for 8GB RAM |
| Embeddings | nomic-embed-text (via Ollama) | Lightweight, strong on technical text |
| Vector DB | ChromaDB 1.0.12 | Python-native, no extra infrastructure |
| API | FastAPI 0.115.9 | Async, typed, auto-docs |
| Orchestration | Docker Compose | Simple and reproducible |
| CI/CD | GitHub Actions | Automated tests + SSH deploy on push |

## Quick Start

### Prerequisites

- Docker + Docker Compose
- Python 3.11+
- 8GB+ RAM (for Ollama models)
- 20GB+ free disk space

### Run locally

```bash
# 1. Clone
git clone https://github.com/kgbunshin/incident-copilot.git
cd incident-copilot

# 2. Configure
cp .env.example .env
# Edit .env: set API_SECRET_KEY (openssl rand -hex 32)

# 3. Start services
docker compose up -d

# 4. Pull models
docker exec -it sre-rag-ollama ollama pull mistral:7b-instruct-q4_K_M
docker exec -it sre-rag-ollama ollama pull nomic-embed-text

# 5. Verify
curl http://localhost:8080/health
# → {"status":"ok","ollama":"ok","chromadb":"ok"}

# 6. Ingest a runbook
./scripts/ingest_local.sh ./docs/runbook-template.md

# 7. Query
./scripts/query_cli.sh "OOMKilled pod, what are the steps to investigate?"
```

## API Endpoints

| Method | Endpoint | Auth | Description |
|--------|----------|------|-------------|
| GET | `/health` | — | Check Ollama + ChromaDB connectivity |
| GET | `/stats` | key | Collection chunk count + indexed sources |
| POST | `/ingest/file` | key | Upload `.md`, `.txt`, `.json`, `.log` |
| POST | `/ingest/alert` | key | Alertmanager webhook payload |
| POST | `/query` | key | Natural language RAG query |
| GET | `/query/similar` | key | Find similar chunks by text |

All authenticated endpoints require the `X-API-Key` header.

### Example query

```bash
curl -X POST http://localhost:8080/query \
  -H "Content-Type: application/json" \
  -H "X-API-Key: your-api-key" \
  -d '{"question": "payment-service OOMKilled, what should I do?"}'
```

```json
{
  "answer": "Based on a similar incident, the likely cause is a memory leak...",
  "sources": ["postmortem-2024-01-14.md", "runbook-oom-k8s.md"],
  "confidence": 0.87
}
```

## Claude Code MCP Integration

Incident Copilot exposes an MCP server so you can query your knowledge base directly from Claude Code.

```bash
# Copy the example and configure
cp .mcp.json.example .mcp.json
# Edit .mcp.json: set SRE_API_URL and SRE_API_KEY
```

Available tools inside Claude Code:

| Tool | Description |
|------|-------------|
| `sre_query` | Natural language query → LLM answer with sources |
| `sre_ingest` | Ingest text content into the knowledge base |
| `sre_stats` | Show total chunks and indexed sources |

## Alertmanager Integration

To automatically ingest alerts from your Kubernetes cluster, add this receiver to your `alertmanager.yml`:

```yaml
receivers:
  - name: incident-copilot
    webhook_configs:
      - url: "http://<VPS2-IP>:8080/ingest/alert"
        send_resolved: true
        http_config:
          headers:
            X-Api-Key: "<your-api-key>"
```

## Security

Three layers of protection on the production API:

1. **Firewall** — port 8080 restricted to trusted IPs only (`TRUSTED_IPS` env var)
2. **IP allowlist middleware** — application-level 403 for untrusted sources
3. **API Key** — `X-API-Key` header required on all endpoints

Ollama (`:11434`) and ChromaDB (`:8000`) are bound to `localhost` only and never exposed externally.
